Architectural Report: Security, Scalability, and GDPR Compliance of a RAG System for Dance Schools in the Google Cloud Platform Ecosystem
Introduction to System Architecture and Business Context
Designing modern Software as a Service (SaaS) systems dedicated to specific industries, such as Polish dance schools, requires a harmonious integration of advanced artificial intelligence mechanisms with rigorous personal data protection requirements. In the discussed architecture, the system relies on Retrieval-Augmented Generation (RAG) technology, which aims to provide personalized, contextual answers to mobile app users on Android and iOS platforms. This architecture assumes that the single source of truth consists of collections of text documents, regulations, procedures, and educational materials stored in specially designated "Wiki" folders within the Google Workspace ecosystem, specifically on Google Drive.

From a data engineering perspective, this process involves asynchronous content synchronization, breaking it down into smaller fragments (chunking), and transforming it into high-dimensional embeddings using models provided by the Vertex AI service, such as text-embedding-005 or text-multilingual-embedding-002. Subsequently, these semantic knowledge representations are persistently stored in a Google Cloud SQL relational database running the PostgreSQL engine, enriched with the pgvector extension. When a mobile app user asks a question, the system converts their query into a vector and performs a nearest-neighbor search operation in the database to retrieve the most relevant document fragments, which then feed into a generative large language model (LLM) to formulate a precise answer.   

Building such a data pipeline in 2025-2026 places the burden on system architects to meet three fundamental challenges. First, vector processing and storage in the cloud must be fully compliant with the European General Data Protection Regulation (GDPR), which means an absolute prohibition of unauthorized transborder transfers of Personally Identifiable Information (PII) to the United States. Second, the database infrastructure must guarantee sub-millisecond search times so that mobile applications run smoothly, while avoiding Out of Memory errors during operations on high-dimensional index graphs. Third, the access of automated synchronization scripts to the Google Drive API must be restricted using the most stringent Identity and Access Management (IAM) policies and network mechanisms such as VPC Service Controls to prevent privilege escalation in the event of a potential attack.

This report provides a comprehensive, analytical decomposition of these three areas. This document was prepared for software engineers, cloud architects, and Chief Information Security Officers (CISOs), offering not only a review of technical documentation but primarily second- and third-order conclusions based on the latest market benchmarks and analyses of AI system security incidents.

GDPR Compliance, Data Residency, and Privacy Architecture in Vertex AI
The primary legal challenge for the discussed SaaS system is ensuring that sensitive data of students, instructors, or financial information of dance schools will not be subject to cross-border transfer outside the European Economic Area (EEA). In a cloud services-based architecture, these guarantees must be considered on multiple levels: the physical location of servers, machine learning (ML processing) policies, and the deep mathematical nature of the vectors themselves.

European Regions vs. Processing Guarantees in Vertex AI
Google Cloud Platform clearly separates the concept of Data-at-Rest location from ML Processing location. While storing data in a Cloud SQL relational database has long guaranteed physical disk residency in a selected region, Vertex AI, being a complex, distributed ecosystem of accelerators (GPU/TPU), requires more meticulous analysis.

In 2025-2026, Google introduced rigorous commitments regarding data residency for AI processing, but these commitments cover only select regions in the United States and the European Union. Choosing a region such as Eemshaven in the Netherlands (europe-west4) or St. Ghislain in Belgium (europe-west1) is a crucial architectural decision. Utilizing endpoints located in these specific regions ensures that queries (prompts) containing dance school documents directed to models such as the gemini-embedding or text-embedding family are processed during ML inference strictly within the borders of these territories.   

This architecture guarantees that data is not sent back to central nodes in the US for computation. This means fulfilling the basic GDPR requirement regarding the territoriality of processing. However, selecting a multi-region setup, for example, eu-multiregion, means that although the data will not leave Europe, the load balancer mechanism can dynamically route inference tasks among all available data centers in the EU. While legally acceptable, this can generate minimal latency and complicate auditability for highly rigorous security analyses. For this reason, in systems demanding maximum predictability, "region pinning" directly to a single specified region, such as europe-west4, is recommended, as it features extensive accelerator availability for handling Vertex AI workloads and full support for the Agent Platform architecture.   

It is worth noting that relying solely on region selection does not constitute a fully sovereign solution in light of the US CLOUD Act, which theoretically allows American agencies to request access to data stored by American corporations on foreign servers. Nevertheless, the standard Data Processing Addendum (DPA) agreement with Google Cloud, supported by mechanisms such as Sovereign Controls delivered in partnership with T-Systems (guaranteeing infrastructure management exclusively by EU citizens and an external Security Operations Center), is considered fully sufficient for private SaaS-class systems in 2026.   

The Shadow Data Problem in Embeddings
A significantly more serious and often overlooked architectural aspect is the nature of the vectors themselves stored in the Cloud SQL service using pgvector. Programmers often mistakenly compare vector embeddings to cryptographic hash functions, believing the process is strictly one-way and thereby completely anonymizes textual data. From this flawed assumption, architects conclude that once documents are transformed from Google Drive into vectors and placed in the database, the database no longer processes Personally Identifiable Information (PII).

Modern research in the field of Data Security Posture Management (DSPM) for AI systems has conclusively disproven this thesis. Vector embeddings are, in reality, highly dense mathematical representations of semantic meaning. Studies, such as those conducted by researchers at Princeton University in 2023 and subsequent years, have demonstrated the massive effectiveness of embedding inversion attacks. This attack involves retrieving an "unreadable" string of numbers from a vector database and repeatedly feeding candidate texts into the same model (e.g., text-embedding-005) to iteratively minimize the vector difference. This results in the ability to reconstruct original text containing surnames, students' health problems, or confidential notes with an accuracy ranging from 78% to over 90%.

Recommended Data Pipeline for GDPR Compliance
To guarantee that the Polish SaaS for dance schools carries zero risk of PII exfiltration and meets regulatory guidelines, simply selecting a European region in Vertex AI is necessary but insufficient. The RAG architecture must implement a "Privacy by Design" strategy at the pipeline processing level.

Proper organization of such a pipeline relies on using an edge redaction mechanism. When an authorized script fetches file content from Google Drive, the data must not be sent directly to the Vertex AI model. Instead, the pipeline should process the raw text through a lightweight Data Loss Prevention (DLP) system or a Named Entity Recognition (NER) model running locally within the GCP perimeter. The function of this step is tokenization. The phrase "Instructor Jan Kowalski complains about knee pain" must be translated into "Instructor [Person_1] complains about [Ailment_1]". Only this redacted text is sent to the europe-west4 region to the Vertex AI service for vectorization.   

This organized process solves two massive problems. On one hand, it immunizes the system against vector database inversion attacks (because the vectors no longer contain PII). On the other hand, it categorically resolves the issue of Article 17 of the GDPR, namely the Right to be Forgotten. If raw student data permeated into the vectors and database, deleting their trace from the system would require manually, complexly, and computationally expensively rebuilding the entire RAG database. Using mapping tables and tokens solves this systemically – the administrator deletes the token linking key in the relational database, instantly and permanently anonymizing all vectors associated with that person.   

Performance and Memory Management for the pgvector Extension in Cloud SQL
In the described SaaS solution, the Cloud SQL database serves a dual role: it functions as a classic relational database storing metadata about users, subscriptions, and facilities, while also (thanks to pgvector) acting as a native vector search engine. The decision to use a PostgreSQL extension instead of dedicated vector solutions (so-called zero-ops, like Pinecone or Qdrant) reduces operational complexity, preserves ACID transactionality, simplifies backup procedures, and allows hybrid queries within a single SQL transaction.   

From a business perspective, the assumed database size of tens of thousands of vectors (e.g., 10,000 to 90,000 document chunks) represents a "micro" scale dataset. The pgvector extension has been heavily optimized in 2025-2026 and successfully competes with native tools in clusters numbering tens of millions of vectors. Nevertheless, due to the specifics of indexing algorithms in high-dimensional vector space, a lack of solid PostgreSQL database configuration will lead to sudden performance drops and memory shortages, as evidenced by incidents reported by architects forced to diagnose errors such as could not resize shared memory segment.   

Vector Math and RAM Consumption
To estimate memory allocation, the dimensionality of data structures generated by Vertex AI must be analyzed. In 2026, a widely used model for general-purpose queries in Polish and English is text-embedding-005 (and its multilingual equivalents), which typically outputs vectors of 768 dimensions (with the possibility of reduction thanks to MRL - Matryoshka Representation Learning).   

In the standard PostgreSQL configuration (FLOAT32), a single dimension costs 4 bytes of space. This means that a 768-dimension vector will consume about 3076 bytes of disk and RAM space. For a collection of 90,000 text chunks, raw vectors will take up just over 276 MB. This size represents a negligible fraction of memory even in the smallest Enterprise-class Cloud SQL instances. However, the raw vector itself does not provide fast search capabilities. To reduce latency from seconds to single milliseconds—required by modern Android and iOS mobile apps—it is strictly necessary to use an Approximate Nearest Neighbor (ANN) index.   

The default and strictly recommended indexing algorithm in a production environment is HNSW (Hierarchical Navigable Small World). Unlike the older and less precise IVFFlat, HNSW operates on a multi-layer graph structure. Despite its higher precision, HNSW carries a memory overhead – the index itself can be 1.5 to even 3 times larger than the raw data table because it must store metadata about the edges connecting neighboring vector points across all layers.   

Even with the increased size, the total memory footprint (data + index) will not exceed 1 GB of RAM. This means that the entire "Wiki" dataset can, and inherently should, constantly reside in the database's fast cache memory.

Optimizing PostgreSQL Operational Parameters
The stability of HNSW on Cloud SQL strongly depends on several critical database environment parameters. Managing them blindly leads to RAG performance degradation.

Tuning shared_buffers: This parameter determines the main PostgreSQL cache, which holds data blocks before flushing to the OS layer. In Google Cloud SQL instances, this is typically configured at around 25% of the server's physical RAM. It is a mistake to drastically increase this value (e.g., to 80%) because database algorithms benefit from a symbiosis with the OS cache, and an excess of dedicated memory for Postgres can result in forced evictions to swap or Out of Memory errors. Cache memory of 2 GB to 4 GB will be more than sufficient for the calculated data density.   

maintenance_work_mem Configuration and Multi-Access Segment Errors: This parameter governs index building. Due to the graph structure of HNSW, the build process requires loading the data entirely into working memory after allocating space for the graph. If the size of the rebuilt index exceeds the maintenance_work_mem limit, Postgres will try to spill the operation to disk (disk spilling). This leads to generating extremely heavy system logs (hnsw graph no longer fits into maintenance_work_mem), after which the index build time multiplies, and the generated graph itself has lower precision. Setting this parameter to 1-2 GB for the duration of the asynchronous vectorization phase from the "Wiki" folders entirely eliminates this phenomenon.
Additionally, numerous analytical reports from engineers using GCP (in 2024-2025) pointed to an issue occurring during index builds: ERROR: could not resize shared memory segment. This was closely tied to the parallel index builds system introduced in newer pgvector versions. In environments like Cloud SQL, the shared memory limit is strictly controlled by the hypervisor. The industry "golden rule" remedy is to temporarily lower or zero out the max_parallel_maintenance_workers = 0 parameter during initial index builds, or to properly adjust the memory relative to the number of allocated worker cores.   

Size Optimization via halfvec Quantization: An effective, proactive architectural step when using Vertex AI models is storing the vector as a 16-bit floating-point number (scalar quantization). The halfvec(768) column type cuts the disk I/O load in half, reducing the size from 3 KB to approx. 1.5 KB per row. The bandwidth savings from SSD to memory, combined with the ability to pack more pages into the shared_buffers area, has a dramatic impact on maintaining consistent, sub-50ms latencies required for mobile environments. Despite lowered precision, this negatively impacts semantic search recall only on a micro scale, which is negligible for Q&A systems.   

Data Isolation Using Row-Level Security in a Multi-Tenant Architecture
The SaaS application serving Polish dance schools is inherently a Multi-Tenant environment. One of the most common design failure points in such architectures is search context leakage – a query from School A must never interact with the vector space placement of School B's documents.

The traditional programming approach, based on appending a WHERE tenant_id = X clause in the application code to every database query, exposes the company to massive risk of human error; bypassing a single application layer would permanently mix prompts in the RAG system, leading in the worst-case scenario to revealing, for instance, rent amounts or competitor student data.   

PostgreSQL architects indicate that the best practice for 2026 is isolating data using the Row-Level Security (RLS) mechanism. Implementing RLS offloads the application code, guaranteeing that the isolation filter is applied at the lowest tier of the database. The authorization mechanism sets the current user's context in the transaction, and RLS policies (e.g., CREATE POLICY tenant_isolation ON wiki_docs USING (tenant_id = current_setting('app.current_tenant')::UUID);) execute rigorous divisions.

Implementing shared tables and RLS in pgvector brings another nuanced technical problem: spatial filtering (Post-Filtering). When the HNSW algorithm navigates the graph during a K-Nearest Neighbors (k-NN) query, it focuses solely on the cosine distance of mathematical dimensions. Only after gathering potential candidates does the Postgres relational engine eliminate rows that do not meet the WHERE clause rules (including the RLS imposed for a specific tenant_id). In classic versions, this could lead to a phenomenon where the system rejected 9 out of 10 candidates, narrowing the RAG context with a drastic drop in answer precision for language models.   

A highly recommended remedy for this, available in pgvector versions after 0.8, is forcing so-called iterative scanning in session parameters (SET hnsw.iterative_scan = strict_order;). This mechanism instructs the database to search until it saturates the candidate pool to the required size despite strict metadata filters (imposed by the system's RLS layer), thereby saving the overall SaaS application performance with a guarantee of uninterrupted precision continuity.   

For small databases with low data density per tenant (e.g., 100 documents from one school), an approach called server-side partitioning by tenant_id and skipping the HNSW index in favor of sequential scans with a perfect 100% exact match rate is often recommended. However, due to the database's expansion dynamics, the combination of HNSW with Iterative Scan constitutes the most stable production option, offering uniform latency characteristics up to 50 milliseconds regardless of data growth for a single facility.   

Designing Network and Logical Access Architecture (IAM, Google Drive API, VPC Service Controls)
Integrating data scattered across various spaces within Google Cloud constitutes an open attack surface, particularly when the environment operationally processes and assimilates raw text data from Google Workspace into a dense relational vector database. A typical, yet fatally flawed approach by engineers in prototype projects is entrusting synchronization scripts with global credentials. The evolution of Google Cloud cyber-protection mandates precise isolation of scripts acquiring content from "Wiki" folders, utilizing deep layered structures: authorization and network traffic security.

Folder-Level Access Authorization "Wiki" - Google Drive API
The OAuth 2.0 protocol and Google Drive API architecture impose a fundamental and often painful limitation for developers: the scope allocation system (e.g., the https://www.googleapis.com/auth/drive scope) cannot apply per-folder barriers at the token generation stage. Granting global scope makes the authorized process "visually" entitled to read any file that the designated user shares with the technical account. A popular alternative, narrower scope, drive.file, also proves dysfunctional for SaaS sync systems – it allows visibility only into documents that the application (script) itself opened or created. For an application meant to run in the background and passively sync content created by dance school staff in the Google Docs interface, this is fundamentally useless, by definition resulting in a 404 Not Found status for reading other people's file structures from a designated "Wiki" folder.   

The only effective, secure, architectural way to grant permissions to an automated script (running, for example, in Cloud Run or as a Cloud Function) without involving leak-prone Domain-Wide Delegation (DWD), is assimilating an authorization mechanism based on Service Accounts combined with the institution of Shared Drives.   

The construction of a proper authorization layer should consist of the following steps:

Service Account Lifecycle Management: The script receives technical identities assigned via a Service Account permanently locked to Workload Identity Federation credentials. At the organizational level, manual generation of static .json certificate files must be explicitly prohibited by activating the iam.disableServiceAccountKeyCreation policy. This defense in depth practice secures the environment in case of repository leaks, preventing credential exfiltration because the process functions exclusively through a system of ephemeral token exchange calls to the authentication platform from a designated cloud area.   

Organizing Shared Drives Spaces: Dance school creators do not establish shared folders on their private virtual drives (because these inherit ownership and are hard to rigorously isolate for bots). The Workspace cloud administrator configures native areas classified as Shared Drives – e.g., "Wiki Hip-Hop School Warsaw". Files pinned to Shared Drives are stripped of private attribution, securing pipeline continuity in cases such as an employee leaving, who would otherwise lose owner rights.   

Granting Rights and the fileOrganizer Role: Our service account's email identifier is injected via the Google Workspace panel (or via an assigned admin role) as a Member exclusively on a specific Shared Drive. Any exploration attempt going beyond the permitted Shared Drive will hit an Access Denied block.
If the script fetching vector knowledge only needs to read changes, a precise and secure reader role is assigned. In situations where the process must also signal status in metadata or apply structural modifications, the narrowed fileOrganizer role is used (which prevents entirely modifying the membership structure – an action secured by the overarching organizer tier).   

For the script to correctly decode the shared structure when requesting a full array of documents from the drive.googleapis.com API, the loop querying via REST/GraphQL must be armed with the system declaration supportsAllDrives=true (warning the API that the client is interacting with an isolated Shared Drive sphere) and, additionally, includeItemsFromAllDrives=true in the list query parameters.   

Limiting the Blast Radius with VPC Service Controls (VPC-SC)
Failing to fortify the communication pipeline at the system firewall level leaves the database infrastructure exposed to lateral movement attacks. In the era of ransomware escalation and attacks involving silent data exfiltration to unrelated cloud accounts controlled by an intruder, the IAM mechanism alone does not protect the Postgres Cloud SQL database. The solution at the Google Control Plane level is the Virtual Private Cloud Service Controls (VPC-SC) platform.

VPC-SC allows constructing an invisible corridor for the transfer of raw data retrieved from Drive. Based on this, the document is sent to aiplatform.googleapis.com model endpoints (Vertex AI in the European location), and after appropriate transformation, into the high-dimensional database (via the sqladmin.googleapis.com API, which secures the SQL cluster itself).

A model VPC-SC implementation for this SaaS will include two stages: building the Perimeter and configuring Ingress/Egress systems.

Stage 1: Building the Security Perimeter for Trusted Endpoints
The project center is enclosed in a hermetic area bound by a designated Service Perimeter. By configuring the environment using commands or tools like Access Context Manager, we introduce strict barriers forbidding any information exchange with the internet on specified service types.
The foundational RAG services are added to the restricted-services section: the aforementioned database API and AI infrastructure API. Placing them under the VPC-SC umbrella immediately blocks unapproved systemic connections.   

Stage 2: Defining Ingress and Egress Rules (Opening Target Channels)
The perimeter by definition also chokes operations of compute scripts located in the GCP cloud, preventing them from connecting to the rest of the public network (including Workspace). When a script attempts to download data from the Google Drive interface ("Wiki"), the security system logs an anomaly, generates RESOURCES_NOT_IN_SAME_SERVICE_PERIMETER alarm blocks, interrupts the application cycle, displays a message on the Security Operations console, and immediately logs an anonymized token for the VPC Service Controls Violation Analyzer platform.

During this period, the blocking engine restrains from brutally severing streams, emitting silent warnings in the Cloud Audit Logs section instead, thus protecting organizations from accidental infrastructure disconnection. Identifying false alarms guarantees that the production switch to enforcing perimeter laws will brick up the gates for attackers while providing a safe "window to the world" for vectorization pipelines.   

Summary and Strategic Architectural Recommendations
Designing a scalable, lightning-fast system based on a RAG architecture targeted at a professional and specific demographic entails enormous responsibility and hinges on technological details. In 2026, one cannot rely solely on "default" configurations offered by automatic systems – operating systems effectively and legally on a mass scale requires awareness of processes hidden deep beneath cloud layers.

The key decisions around which the architecture of the discussed system should revolve fall into the following three theses drawn from the analyses conducted:

GDPR Compliance (Data Residency and PII Transformation): The physical residency of model servers (europe-west4, europe-west1) used for vectorization in the Vertex AI computing infrastructure indisputably fortifies the system in terms of ISO and GDPR certification regarding ML data flows. However, vectorizing text hiding identities (PII) carries a critical compliance payload due to the phenomenon of inversion strikes with documented leaks. Thus, it is essential to operate with layered anonymization and tokenization (supported by DSPM policies) so that texts transmitted to the Vertex AI algorithm operate exclusively on substitute pseudonyms or universal variables.   

Optimizing PostgreSQL with pgvector for Mobile Apps: Despite a moderate data volume (on a macro scale, 10,000 to 90,000 entries is small, around a few hundred megabytes per table), a precise and memory-safe response will determine whether the mobile app for dance schools runs without stuttering. Applying halfvec(768) quantization cuts disk density in half while preserving maximum throughput. Simultaneously, an imposed per-school protection layer, based on a structural Row-Level Security (RLS) system reinforced by HNSW Iterative Scan, will prevent the risk of mixing search results between different schools (post-filtering drift). To insure the cloud environment against operational exhaustion and looping during re-indexing, the system absolutely requires manipulating allocations in PostgreSQL by increasing the maintenance_work_mem reservation limit to 1-2 GB on the server space.   

The Fortress and Extra-Perimeter Authorization Channels: Connecting scripts and platforms fetching documents on the frontend (Google Drive) without the ability to cast structured boundaries generates vulnerabilities. Rejecting general authentication credentials in favor of delegated roles assigned to strictly specified Shared Drives using organizer roles (fileOrganizer) builds a watertight permissions network allowing access only to targeted "Wiki" knowledge. In the IT layer, the system is subject to an overarching control layer organized by a locked Perimeter on VPC Service Controls, from where egress is impossible except for a certified account aiming through a cracked Egress throttle toward Drive operations, thereby drastically reducing potential escalation avenues targeting the services. Securing compute instances with Workload Identity insures the SaaS core itself against the loss of defined privileges.
